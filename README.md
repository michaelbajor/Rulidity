# Rulidity

A Rust eDSL that compiles to EVM bytecode. It is heavily inspired by Substrate's ink!.

The general idea is to have Rust compiler check for types and mutability and then have it compiled to EVM bytecode.

Another approach would be to create an LLVM target of EVM, but that would be a huuuugeeee pain due to architectural differences.

No-one asked for Rulidity and nowadays no-one really needs it. But I thought it might be a fun excercise to do.

## How to use this?

Well, the Counter would look like this:

```rust
#[rulidity::contract]
mod counter {
    use rulidity::prelude::*;

    #[storage]
    struct Counter {
        count: U256,
    }

    impl Counter {
        #[constructor]
        fn new(&mut self, start: U256) {
            self.count = start;
        }

        #[external]
        fn increment(&mut self) {
            self.count = self.count + U256::from(1);
        }

        #[external]
        fn add(&mut self, val: U256) {
            self.count = self.count + val;
        }

        #[external]
        fn get(&self) -> U256 {
            self.count
        }
    }
}
```

A more concrete (and fun, but also longer) example can be found in `examples/erc20.rs` file.

## Foundry integration

Emitting a bytecode wouldn't be fun if we couldn't deploy it. Good news - you can!

Sooo, using `examples/erc20.rs` file, when you run it, it will generate an `ERC20.bin` and `ERC20.abi.json` files. Copy them somewhere into your Foundry project.

Now, you can use `cast` to parse the ABI file and generate an actual Solidity interface. You'd do it more-or-less like so:

```bash
cast interface ./artifacts/ERC20.abi.json --name IERC20 --output test/IERC20.sol
```

You might need to add following into the `foundry.toml` file:

```toml
[profile.default]
fs_permissions = [{ access = "read", path = "./" }]
```

Now, you can create a test file in your foundry and add:

```solidity
pragma solidity ^0.8.13;

import {Test, console} from "forge-std/Test.sol";
import {IERC20} from "./IERC20.sol"; // generated by cast

contract ERC20 is Test {
    IERC20 token;
    uint256 constant TOTAL_SUPPLY = 1_000_000e18;
    string constant NAME = "TEST_TOKEN";
    string constant SYMBOL = "TST";

    function setUp() public {
        bytes memory creation = vm.parseBytes(
            vm.readFile("./artifacts/ERC20.bin")
        );

        bytes memory deployData = abi.encodePacked(
            creation,
            abi.encode(TOTAL_SUPPLY, NAME, SYMBOL)
        );

        address addr;
        assembly {
            addr := create(0, add(deployData, 0x20), mload(deployData))
        }

        require(addr != address(0), "deploy failed");
        token = IERC20(addr);
    }
}
```

Aaaand that should work! Exemplary tests would be:

```solidity
    function test_supplyWorks() public view {
        assertEq(token.totalSupply(), TOTAL_SUPPLY);
        assertEq(token.balanceOf(address(this)), TOTAL_SUPPLY);
    }

    function test_metaWorks() public view {
        assertEq(token.name(), NAME);
        assertEq(token.symbol(), SYMBOL);
    }

    function test_transferWorks() public {
        address user = address(0x1);
        token.transfer(user, 100);
        assertEq(token.balanceOf(user), 100);
        assertEq(token.balanceOf(address(this)), TOTAL_SUPPLY - 100);
    }

    function test_approveWorks() public {
        address spender = address(0x2);
        token.approve(spender, 10);
        assertEq(token.allowance(address(this), spender), 10);
    }
```

## Limitations

- not all Rust types can be used in functions
- Mappings and Dynamic Arrays are supported, but static Arrays are not
- Only U256, Address and bools are supported from the "classic" Solidity types.
- Support for strings is limited to short ones (< 32 bytes)
- No differentiation between `external` and `public`, i.e. every callable function is considered `external` (as noted by the `#[external]` macro)
- No differentation between `payable` and non-`payable` functions. The ABI generation marks all functions as non-`payable`, however actual implementation does not add attached deposit checks, so technically all are payable (should be an easy fix tho)
- contracts cannot call other contract
